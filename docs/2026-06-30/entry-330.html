# Agent Research Methodology


A general-purpose research skill for any topic. Combines source credibility

tiers, multi-angle search, confidence gating, and a manager quality gate into

a structured workflow. Built on top of the skill-harness loop pattern.


## When to Load This Skill


Load this skill when the user asks you to research any topic that requires

multiple sources, fact-checking, or analysis — stocks, tech, industry trends,

macro economics, geopolitics, company analysis, etc.


## The Research Loop


                    ┌─── early_exit (≥90%)


                    │


GENERATE ─→ VALIDATE ─── standard (50-90%) ─→ REFINE ─→ REGENERATE


    ↑              │                                    │


    │              └─── deep_search (<50%) ───────────────┘


    └──────────────────── REFINE feedback ────────────────┘


### Phase 1: GENERATE — Multi-Angle Collection


Do NOT do a single search. Research needs multiple angles:


# For a stock: AAPL


searches = [


    "AAPL stock price technical analysis 2026",


    "AAPL analyst rating price target June 2026",


    "Apple latest news catalysts market reaction",


    "AAPL earnings revenue EPS growth outlook",


    "Apple competitive position AI strategy"  # different angle


]




# For a tech topic: "AI chip shortage"


searches = [


    "AI chip shortage 2026 market impact",


    "NVIDIA TSMC semiconductor supply chain",


    "AI chip alternatives China export controls",


    "Data center GPU demand forecast 2026 2027",


]


Rules:

  • Minimum 3 angles — never 1 search only
  • If the first 3 searches return conflicting data, add 2 more to resolve
  • Use different sources per search (not all Yahoo Finance)
  • Use web_extract/browser for deep-dive on key sources

    • ### Phase 2: VALIDATE — Source Credibility + Confidence Gate


    #### Source Credibility System


    Tag every source with its tier. The reader sees this immediately.


    | Tier | Type | Examples | Trust |

    |------|------|----------|-------|

    | Tier-1 | Peer-reviewed, official filings | SEC 10-K/10-Q, Nature, Journal of Finance, IEEE, IMF, World Bank, Federal Reserve | ✅ Primary |

    | Tier-2 | Major research, reputable analysts | McKinsey, BCG, Bain, Gartner, Forrester, IDC, Goldman Sachs, BofA, Bloomberg Terminal | ✅ Strong |

    | Tier-3 | Quality journalism | FT, WSJ, Bloomberg, Reuters, The Economist, CNBC (selective), NYT, WaPo | ✅ Reliable |

    | Tier-4 | Trade press, secondary | TechCrunch, The Verge, Seeking Alpha, MarketBeat, Yahoo Finance articles, Barrons | ⚠️ Needs corroboration |

    | 🚫 Prohibited | Unattributed, anonymous | Reddit, anonymous blogs, Wikipedia as primary, X/Twitter unverified | ❌ Never cite alone |


    Corroboration rule: Tier-4 claims MUST be backed by at least one Tier-2/3

    source. Two Tier-4 sources agreeing is still not enough — find a stronger source.


    #### Confidence Gate


    After collecting data, self-rate your confidence per section:


    Section: [Technical Analysis]
    

    Source: Yahoo Finance Chart API (Tier-3) + TradingView (Tier-3)
    

    Key claim: AAPL support at $287
    

    Confidence: 85% ← price from live API, no interpretation
    

    

    Section: [Analyst Sentiment]
    

    Source: MarketBeat price targets page (Tier-4)
    

    Key claim: Avg PT $314.59, 24 Buy / 10 Hold / 1 Sell
    

    Confidence: 70% ← MarketBeat is Tier-4, need cross-ref


    Actions by confidence level:


    | Confidence | Mode | Action |

    |------------|------|--------|

    | ≥90% | early_exit | Accept as-is, no refine needed |

    | 50-90% | standard | Do 1 round of REFINE: search for corroboration |

    | <50% | deep_search | Reframe the question entirely, 2+ new angles, re-search |


    #### Manager Quality Gate (Pre-Ship)


    Before delivering any research output, run this gate:


    For EVERY claim with a specific number, date, or attribution:
    

      ✅ [HALLUCINATION RISK] — claim has no source, or contradicts source?
    

      ✅ [CITATION NEEDED] — plausible but unsourced?
    

      ✅ [STALE DATA] — source older than 6 months?
    

      ✅ [CONFLICT] — source has financial interest? (e.g. bank covering own stock)
    

      ✅ [OUTDATED] — data superseded by newer information?
    

    

    For the report as a whole:
    

      ✅ Are all numbers sourced?
    

      ✅ Are all Tier-4 claims corroborated by Tier-2/3?
    

      ✅ Confidence score ≥ 70% overall?


    Nothing ships without a Manager pass. If any flag is raised, block delivery

    and enter REFINE phase.


    ### Phase 3: REFINE — Targeted Improvement


    Based on which issues the Manager Gate found, pick the right refinement strategy:


    | Issue | Strategy |

    |-------|----------|

    | [HALLUCINATION RISK] | Re-fetch the source, quote exact text, re-derive |

    | [CITATION NEEDED] | 1 specific search for the missing citation |

    | [STALE DATA] | Re-fetch with current date range filter |

    | [CONFLICT] | Find a neutral source for the same data point |

    | Confidence < 50% (deep_search) | Entirely new search angles, not just rephrasing |


    Reflexion note: Before refining, note what went wrong:

    Reflexion: My first search for "AAPL analyst targets" returned MarketBeat
    

    (Tier-4) which gave avg PT $314. But I didn't verify against TipRanks or
    

    Bloomberg. For the refine pass, I'll search "AAPL analyst price target
    

    June 2026 consensus" to find a Tier-3 source.

    ### Research Output Checklist


    BEFORE DELIVERING, CONFIRM:
    

    

    □ Minimum 3 search angles used
    

    □ All claims tagged with source tier
    

    □ No Tier-4 claims used without corroboration
    

    □ Numbers triple-checked against source

    □ Sources table included
    

    □ Risk factors acknowledged (not just bull case)

    ## Cron Job Integration


    The agent-research methodology can power scheduled cron jobs, not just

    ad-hoc research. This section covers how to design cron job prompts that

    follow the methodology.


    ### Pattern 1: Research Agent (e.g. sim-research)


    Full methodology for every tick: multi-angle search, source tiers, confidence

    gate, Manager Gate. 1 stock per tick (context guard) — write enriched JSON

    research files to disk.


    Skills loading:

    cronjob action=create name="sim-research" skills=[agent-research] ...


    Prompt structure:

  • "You are a stock research agent following the agent-research methodology."
  • Lock: "1 stock per tick. NEVER process 2."
  • Context guard: stop early if conversation grows long
  • Explicit phases: READ → GENERATE (3+ angles) → VALIDATE (source tiers +

    • confidence gate) → REFINE (if needed) → ACCEPT (write enriched research file

    with overall_confidence, manager_gate, sources_tiers, angles_used)

  • Output: brief one-liner, research data goes to file (deliver=local)

    • ### Pattern 2: Dual-Mode Monitor (e.g. Stock Monitor Watchlist)


    Lightweight mode for normal ticks; research mode triggered by anomaly.

    ⚠️ This pattern is PRONE to DeepSeek 180s streaming timeout / Broken pipe

    when the script output is large AND the agent tries to further enrich via web

    searches. For data-heavy monitors (prices + options flow), prefer Pattern 4

    (no_agent=true) instead.


    Skills loading:

    ``cronjob action=create name="Stock Monitor" skills=[agent-research] ...

    

    Dual-mode switch:
    

  • Normal mode (no >5% move): compact price report, no research tool calls
    • 

  • Research mode (>5% move detected): activate GENERATE phase — min 2 angles
    • 

      (news catalyst + sector peer), source tier tag, confidence gate, Manager Gate
    

  • Keep research output compact (1-2 sentences per anomaly) to avoid context bloat
    • 

    

    When this pattern works:
    

  • Simple data-formatting (script output → 1-2 sentence summary)
    • 

  • Single source of data (prices only, no options flow or multiple enrichment passes)
    • 

  • Web tools are DISABLED in enabled_toolsets (no temptation to browse)
    • 

  • Report fits in <1000 chars / <2s of streaming
    • 

    

    When to switch to Pattern 4 (no_agent=true):
    

  • Script collects HEAVY data (prices + options + sector data)
    • 

  • Any enrichment step (browser, web_search, delegate_task) could be triggered
    • 

  • Job has ever failed with [Errno 32] Broken pipe
    • 

  • Report is pure formatting with no reasoning needed (sort, format, deliver)
    • 

    

    ### Pattern 3: Technical + Fundamental Hybrid (e.g. AAPL+AMZN Technical Monitor)
    

    

    Combines browser-based TA extraction with agent-research quality gates.
    

    

    Skills loading:

    cronjob action=create name="AAPL+AMZN Technical" skills=[agent-research, us-market-data] ...

    

    Prompt structure:
    

  • Each technical claim tagged with source tier (TradingView oscillators = T-3,
    • 

      TradingView pivots = T-3, Yahoo Chart API = T-3)
    

  • Confidence score per section (Technical: XX%, Analyst sentiment: XX%)
    • 

  • Manager Gate pre-ship: [HALLUCINATION RISK] / [CITATION NEEDED] /
    • 

      [STALE DATA] / [CONFLICT]
    

  • Optional: web_search for fundamental catalysts when notable moves detected
    • 

  • Sources table at bottom of report
    • 

    

    ### Pattern 4: no_agent=true Data Monitor (Broken Pipe Proof)
    

    

    For data-heavy cron jobs that collect structured data and need formatting
    

    (prices, options flow, sector data), use no_agent=true with a Python script
    

    that does BOTH collection AND formatting. No LLM agent = no streaming timeout.
    

    

    Use when:
    

  • Script collects multiple data sources (prices + options + sector data)
    • 

  • Any enrichment step could be triggered (web_search/browser)
    • 

  • Job has ever failed with [Errno 32] Broken pipe
    • 

  • Report is pure formatting (sort by → flag anomalies → format emoji)
    • 

  • Output goes to Telegram (short, concise, emoji-rich)
    • 

    

    How to build:
    

    

    Single Python script at ~/.hermes/scripts/<name>.py that:
    

  • 1. Runs data-collection sub-scripts via subprocess.run([sys.executable, path], ...)
    • 

  • 2. Parses JSON output from each
    • 

  • 3. Formats into plain text (no markdown tables — Telegram has no table support)
    • 

  • 4. Outputs to stdout → delivered verbatim
    • 

    

    Cron job config:
    cronjob action=update job_id="..." no_agent=true script="stock_monitor_report.py"
    Note: prompt and skills are ignored when no_agent=true.
    

    

    Report format conventions (Telegram delivery):
    

  • Use bold for tickers and significant numbers
    • 

  • Emoji for direction: 🚀/📈 big up, 💀/📉 big down, ☎️ call, 🛡️ put
    • 

  • Sort by abs(change) descending — most important first
    • 

  • Include P&L for portfolio holdings when tracked
    • 

  • Keep under 2000 chars total
    • 

    

    Example: stock_monitor_report.py (working production script)
    python

    ### Cron Job Review Checklist
    

    

    When upgrading existing cron jobs to use agent-research:

    □ Job has agent-research in skills: [agent-research] (not just in prompt text)

    □ Prompt explicitly references methodology phases (GENERATE/VALIDATE/REFINE/ACCEPT)

    □ Source tiers are tagged on every claim

    □ Confidence gate: ≥90% early_exit, 50-90% standard refine, <50% deep_search

    □ Manager Gate runs before delivery

    □ Normal-mode jobs stay lightweight (no unnecessary research tokens)

    □ Research-mode jobs trigger only on anomaly signals (>5% move, pattern break, etc.)

    □ Sources table or summary in output

    □ Risk factors / contradictions acknowledged

    

    ### Pitfalls (Cron Jobs)
    

    

  • ❌ Don't just mention agent-research in prompt text without adding it to
    • 

      skills: [agent-research] — the prompted methodology text is a pale shadow
    

      of the full loaded skill, and the agent won't have access to references/
    

      or the detailed source-tier system.
    

  • ❌ Don't keep cron jobs in always-on research mode — 3+ searches per tick
    • 

      on every 30-minute run will blow through context and DeepSeek's 180s timeout.
    

      Use dual-mode: lightweight default, research only on anomaly.
    

  • ❌ Don't forget to enable web toolset on cron jobs that do research —
    • 

      without enabled_toolsets: ["web", "file", "terminal"], web_search won't work.
    

  • ✅ Chain with domain-specific skills: [agent-research, us-market-data] for
    • 

      stock work, [agent-research, arxiv] for academic research, etc.
    

  • ✅ For technical analysis jobs (AAPL+AMZN Technical Monitor style), chain
    • 

      agent-research + us-market-data — the first provides methodology, the
    

      second provides stock-specific data source references.

    ## Pitfalls
    

    

  • ❌ Don't stop at 1 search — always 3+ angles minimum
    • 

  • ❌ Don't trust Tier-4 sources alone — always corroborate with Tier-2/3
    • 

  • ❌ Don't skip the Manager Gate — it catches the most embarrassing errors
    • 

  • ❌ Don't deliver without a confidence score — the user needs to know how sure you are
    • 

  • ❌ Don't ignore contradictions — risk factors are as important as bullish findings
    • 

  • ❌ Don't use stale data without flagging — tag everything with a date
    • 

  • ❌ Don't rely on a single search backend — web_search and web_extract share the same backend, so both fail together (e.g. 402 credit errors). Have a fallback chain: DuckDuckGo HTML endpoint via curl (html.duckduckgo.com, bypasses CAPTCHA) → browser navigation for JS sites → curl with User-Agent for static pages/APIs → direct URL navigation to known authoritative sources → GitHub API search. See references/search-fallback-chain.md §1b for the DuckDuckGo HTML endpoint technique.
    • 

  • ❌ Don't search when the user shares a screenshot for source verification without first doing image forensics. The image itself (dimensions, metadata, visual style, layout, presence/absence of branding/URLs) often identifies the source class (AI chat vs blog vs documentation) faster and more reliably than searching — especially when all search engines are blocked by CAPTCHA. See references/search-fallback-chain.md §「Image / Screenshot Source Verification」for the full checklist.
    • 

  • ❌ Don't hammer the Wikipedia API — Multiple rapid action=parse calls trigger HTTP 429 (rate limit). Always insert time.sleep(1) between requests. Fetch sections sequentially, not in parallel. For lightweight checks, use the mobile page (en.m.wikipedia.org) with curl instead.
    • 

  • ❌ Don't run LLM-agent cron jobs on heavy data scripts — script output plus web enrichment causes context bloat → DeepSeek streaming times out at 180s → [Errno 32] Broken pipe → job enters a death spiral where every tick fails. Fix: convert to no_agent=true with a Python script that both collects and formats (see Pattern 4 above). This is the #1 cause of cron job failures in this Hermes setup.
    • 

  • ❌ Don't use pre-event draw/seeding tables to list current tournament teams — once an event has started (World Cup, Euros, etc.), the draw pots table on Wikipedia still shows "UEFA Path winner" placeholder entries that were resolved long ago. Use the group stage tables or squads page instead, which list all actual competing teams. See references/ongoing-event-data-freshness.md.
    • 

  • ✅ Always state the source tier alongside the claim
    • 

  • ✅ Always put the bottom line first (BLUF)
    • 

  • ✅ If two sources disagree, say so, don't cherry-pick
    • 

  • ✅ Reflexion notes improve the next REFINE pass significantly
    • 

  • ✅ For cron jobs: if it's just data → format → deliver (no reasoning needed), use no_agent=true from day one. Only use LLM-agent mode when the job needs real analysis (catalyst synthesis, market context, trade decisions).
    • 

  • ✅ Cost estimation is quick math — 3-5 searches + ~$0.003 DeepSeek tokens per research tick. Not a reason to over-optimize. See references/research-job-cost-estimation.md for exact numbers per backend.
    •
