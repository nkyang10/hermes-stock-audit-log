• ## Cron 作業集成


• 代理研究方法可以為預定的 cron 作業提供支持，而不僅僅是


• 臨時研究。本節介紹如何設計 cron 作業提示


• 遵循方法論。


• ### 模式 1：研究代理（例如 sim-research）


• 每個刻度的完整方法論：多角度搜尋、來源層、置信度


• 門，經理門。每個價格變動 1 隻股票（情境保護）－編寫豐富的 JSON


• 研究文件到磁碟。


• 技能加載：**


• cronjob 操作=建立名稱=“sim-research”技能=[代理研究] ...


• 提示結構：**


• “您是遵循‘代理研究’方法的股票研究代理。”


• 鎖定：“每個價格變動 1 隻股票。切勿處理 2 隻股票。”


• 情境保護：如果對話時間過長，請儘早停止


• 顯式階段：讀取→產生（3+角度）→驗證（來源層+


• 信賴閘）→ 完善（如果需要）→ 接受（撰寫豐富的研究文件


• 與整體信心、經理門、來源層、角度使用）


• 輸出：簡短的一行，研究資料進入檔案（deliver=local）


• ### 模式 2：雙模式監控（例如股票監控觀察清單）


• 普通蜱蟲的輕量級模式；異常觸發的研究模式。


• ⚠️ 此模式容易導致 DeepSeek 180 秒流逾時/管路損壞


• 當腳本輸出很大且代理程式嘗試透過網路進一步豐富時


• 搜尋。對於數據量大的監視器（價格+選擇權流），偏好模式 4


• (no_agent=true) 相反。


• 技能加載：**


•
cronjob 操作=創建名稱=“股票監視器”技能=[代理研究] ...
• 雙模式切換：**


• 正常模式（沒有 >5% 的變動）：緊湊的價格報告，沒有研究工具調用


• 研究模式（偵測到 >5% 的移動）：活化產生階段 — 至少 2 個角度


• （新聞催化劑+業界同業）、源層標籤、置信門、經理門


• 保持研究輸出緊湊（每個異常 1-2 個句子）以避免上下文膨脹


• 當此模式起作用時：**


• 簡單的資料格式化（腳本輸出 → 1-2 句話摘要）


• 單一資料來源（僅價格，無選擇權流或多個豐富通道）


• Web 工具在enabled_toolsets 中被停用（沒有誘惑去瀏覽）


• 報告適合 <1000 個字元/<2 秒的串流傳輸


• 何時切換到模式 4 (no_agent=true):**


• 腳本收集大量數據（價格+選擇權+行業數據）


• 可以觸發任何豐富步驟（瀏覽器、web_search、delegate_task）


• 作業曾因 [Errno 32] 管道損壞而失敗


• 報告是純粹的格式，不需要推理（排序、格式、交付）

• ### 模式 4：no_agent=true 資料監控（斷管證明）


• 對於收集結構化資料並需要格式化的資料密集型 cron 作業


• （價格、期權流、行業數據），將 no_agent=true 與 Python 腳本結合使用


• 它既進行收集又進行格式化。沒有LLM代理=沒有串流超時。


• 使用時間：**


• 腳本收集多個資料來源（價格+選擇權+產業資料）


• 可以觸發任何豐富步驟（web_search/browser）


• 作業曾因 [Errno 32] 管道損壞而失敗


• 報告是純格式的（按→標記異常→格式化表情符號排序）


• 輸出發送至 Telegram（簡短、簡潔、表情符號豐富）


• 如何建構：**


• ~/.hermes/scripts/<name>.py 中的單一 Python 腳本：

• ### 陷阱（Cron 作業）

• ❌ 不要在提示文字中只提及「agent-research」而不將其加入

• skills: [agent-research] — 提示的方法文字是一個蒼白的陰影

• 滿載技能，且代理商將無法存取參考資料/

• 或詳細的源層系統。

• ❌ 不要將 cron 作業保持在始終在線的研究模式下 — 每個週期 3 次以上搜索

• 每運行 30 分鐘就會超出上下文和 DeepSeek 的 180 秒逾時。

• 使用雙模式：輕量級默認，僅研究異常。

• ❌ 不要忘記在進行研究的 cron 作業上啟用「web」工具集 —

• 如果沒有 enabled_toolsets: ["web", "file", "terminal"]，web_search 將無法運作。

• 具有特定領域技能的鏈：“[代理研究，美國市場數據]”

• ### 技能裝備

• 此技能中的GENERATE→VALIDATE→REFINE→ACCEPT循環是一個具體的

• 「技能利用」元技能的實現。每個階段直接映射：

• GENERATE→利用GENERATE功能（多角度採集）

• VALIDATE → 利用 VALIDATE 與特定領域的標準（來源圖層、置信度閘）

• REFINE → 利用 REFINE 制定有針對性的策略（反思、研究）

• ACCEPT → 利用 ACCEPT 結構化輸出格式

• ❌ 不要停留在 1 次搜尋 — 總是至少 3 個以上角度

• ❌ 不要單獨信任第 4 層來源 — 始終與第 2/3 層證實

• ❌ 不要跳過 Manager Gate — 它會捕捉最令人尷尬的錯誤

• ❌ 不要在沒有置信度分數的情況下交付 - 用戶需要知道你有多確定

• ❌ 不要忽視矛盾 — 風險因素與看漲結果同樣重要

• ❌ 不要在沒有標記的情況下使用過時的資料 - 用日期標記所有內容

• ❌ 不要依賴單一搜尋後端 - web_search 和 web_extract 共用相同的後端，因此兩者都會一起失敗（例如 402 信用錯誤）。有一個後備鏈：JS 網站的瀏覽器導航 → 用於靜態頁面/API 的帶有 User-Agent 的「curl」 → 直接 URL 導航到已知的權威來源 → GitHub API 搜尋。請參閱 references/github-api-search.md 和 references/search-fallback-chain.md 以了解完整的層次結構以及每個層級的工作原理。

• ❌ **當使用者共用螢幕截圖以進行來源驗證時，請勿在未先進行影像取證的情況下進行搜尋。圖像本身（尺寸、元資料、視覺風格、佈局、品牌/URL 的存在/不存在）通常比搜尋更快、更可靠地識別來源類別（AI 聊天、部落格、文件）——尤其是當所有搜尋引擎都被驗證碼阻止時。請參閱 references/search-fallback-chain.md §「映像/螢幕截圖來源驗證」以取得完整的清單。

• ❌ 不要攻擊維基百科 API — 多個快速 action=parse 呼叫會觸發 HTTP 429（速率限制）。始終在請求之間插入“time.sleep(1)”。按順序而不是並行地獲取部分。對於輕量級檢查，請使用帶有“curl”的行動頁面（“en.m.wikipedia.org”）。

• ❌ 不要在繁重的資料腳本上執行 LLM 代理程式 cron 作業 — 腳本輸出加上 Web 豐富會導致上下文膨脹 → DeepSeek 流在 180 秒時逾時 → “[Errno 32] 管道損壞” → 更新作業進入死亡螺旋，每次更新作業都會失敗。 修復： 使用可收集和格式化的 Python 腳本轉換為 no_agent=true （請參閱上面的模式 4）。這是 Hermes 設定中 cron 作業失敗的第一大原因。

• ✅ 始終在聲明旁註明來源層

• ✅ 始終把底線放在第一位（BLUF）

• ✅ 如果兩個消息來源不同意，請直說，不要擇優挑選

• ✅ 反思筆記顯著改善下一次 REFINE 通過

• ✅ 對於 cron 作業：如果只是資料 → 格式 → 交付（不需要推理），請從第一天開始使用 no_agent=true。僅當工作需要真實分析（催化劑合成、市場背景、貿易決策）時才使用LLM代理模式。

• ## 工作流程

• ### 第 1 步：閱讀

• 讀取目前時間戳記的ticks/latest.txt。讀取隊列/status.json。找到第一個帶有 Research_status 的程式碼：「pending」。從prices/{ts}/{ticker}.json 讀取其價格檔。閱讀 state/portfolio.json 以了解上下文。

• ### 第 2 步：產生 — 多角度集合

• 搜尋 3 個以上角度。每次搜尋都有不同的來源。最低：

• 角度 1：價格/技術（Yahoo Chart API、TradingView）

• 角度 2：分析師/基本面（雅虎分析頁面、瀏覽器）

• 角度 3：新聞/催化劑（雅虎新聞、瀏覽器）

• 角度 4（可選）：來自 TradingView 的同儕比較

• ### 第 3 步：驗證

• 檢查每個聲明的來源層。每個部分的自評置信度。運行管理器門：

• [幻覺風險] — 聲稱來源不明或與來源相矛盾？

• [需引證] - 看似合理但無來源？

• [陳舊數據] — 超過 6 個月？

• [衝突] — 來源有經濟利益嗎？

• ≥90%→early_exit，寫入研究文件

• 50-90% → 標準：1 輪有針對性的搜尋優化

• <50% → 深度搜尋：重新建置問題，2+ 個新角度

• ### 步驟 4：完善（如果需要）

• 反思筆記：第一遍出了什麼問題？有針對性地重新搜尋特定的缺失資訊。重新驗證信心。

• ### 第 5 步：接受

• 將所有領域的研究寫入文件，包括置信度分數和來源層。更新queue/status.json：設定research_status：「完成」。使用當前價格時間戳更新ticks/latest.txt。
